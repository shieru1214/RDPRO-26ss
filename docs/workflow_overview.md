# Jiaozi — System Workflow Overview

> Presentation / report copy for the end-to-end workflow. Statements describe the **current
> implementation** unless explicitly marked **(Roadmap)**, which denotes a planned next step.

Our overall workflow is systematically divided into three major components:
**Requirement Parsing & Hybrid Retrieval**, **Recipe-Grounded Code Generation**, and
**Automated Validation with DataOps Abstraction**.

---

## 1. Requirement Parsing & Hybrid Retrieval

First, the system captures raw user inputs and model preferences. We leverage an LLM for
feature engineering and requirement parsing, extracting key downstream tags such as
`task_type` (e.g., object detection) and `edge_deployment` from natural language.
Concurrently, the system automatically parses the dataset's metadata to extract crucial
architectural constraints, such as data scale, class imbalance, image-size tier, color mode,
and the total number of classification categories.

In the core retrieval (RAG) stage, vision models are not single networks but **compositions of
multiple modules** (backbone, pretrained weights, head, loss, optimizer) with intricate
topological dependencies and hard routing rules. We therefore **separate hard constraints from
soft ranking**:

**a) Hard routing & compatibility — NetworkX DiGraph.**
A directed graph enforces strict, non-negotiable rules: which pretrained weights belong to which
backbone (`has_pretrained`), which heads/losses are compatible (`compatible_with`), and fixed
integrated architectures that cannot be recombined (`requires`). These act as **filters** and as
the **traversal structure** that assembles a complete, valid configuration. They are *pass/fail*,
not a weighted score.

**b) Hybrid scoring — ranks the candidates that survive the hard constraints.**
- **60% structured, rule-based score** — data-scale match, priority-vs-complexity fit, and
  `preferred_when` bonuses.
- **40% semantic similarity** — match between the model descriptions and the extracted keywords,
  computed with the `all-MiniLM-L6-v2` embedding model.

**c) Deployment-aware downsizing.**
The system is context-aware of the deployment environment. For edge scenarios it tightens the
admissible model-size band and applies a parameter/FLOPs budget filter, automatically triggering
a model-downsizing strategy — e.g., substituting a ResNet-50 with a ResNet-18 — to meet the
strict resource constraints of edge devices.

Ultimately, this stage outputs the **Top-3 candidate configurations** passed down to the next module.
For natural-language classification tasks, each candidate is also enriched with a deterministic
**recipe** and provenance: model/data-aware defaults for image size, learning rate, epochs, and
augmentation.

---

## 2. Recipe-Grounded Code Generation

The next phase is the Code Generation module. Our core design philosophy revolves around a
**"Template-Driven + LLM"** hybrid architecture, explicitly engineered to balance flexibility
with code robustness.

**Foundational `model.py` — LLM generation with a self-correcting loop.**
The model architecture is generated by the LLM to match the chosen configuration. Generation is
wrapped in a **validate-and-retry loop**: the output is checked (valid Python via `ast.parse`,
required `build_model` entry point, rejection of malformed/HTML responses); on failure the error
is fed back to the model for a bounded number of retries; if it still fails, the system falls back
to a deterministic template. This confines the LLM's failure surface and keeps the package executable.
> **(Roadmap)** Evolve this into a full **ReAct-style agent** with explicit reasoning/action steps
> and richer error-detection and self-correction.

**Standardized components — robust pre-defined templates.**
The training loop, `evaluate.py`, data loading, and runners are produced from deterministic
templates. These templates already support context-driven tuning — for example, assigning
**differential learning rates** to the backbone (feature) layer and the head layer when finetuning
a pretrained transformer.

**Hyperparameters — recipe first, explicit overrides respected.**
Hyperparameters are currently supplied by a **rule-based recipe** (learning rate by
strategy/architecture, image size by checkpoint/family/data signals, epochs by data size, and
augmentation by data scale / color mode / constraints), with deterministic template fallbacks.
Module 4 consumes the recipe only when the incoming config does not explicitly override a field,
so manually specified benchmark configs remain stable.
> **(Roadmap)** Add an **AI/LLM hyperparameter proposer** that suggests optimized settings grounded
> in accumulated run history, validated against the rule-based ranges.

Together, this hybrid approach drastically suppresses the LLM's hallucination rate, ensuring the
generated code package is highly stable and executable.

## 3. Execution, Validation, and Skrub DataOps Abstraction

Once the code is generated, instead of a single run we execute `run_experiments.py` to
**sequentially sweep all Top-3 model candidates**, training and evaluating each and emitting a
side-by-side comparison. We also run a **rapid smoke test** (a fast synthetic single-step pass) to
verify pipeline viability and basic model availability before committing to real training.

For controlled algorithm decisions, Module 4 now also supports fold injection and validation
prediction export. The first use case is the loss-imbalance A/B harness, where CE and focal loss
share the exact same stratified 5-fold splits and differ only in the pre-registered loss arm.

To maximize the observability and maintainability of our data pipeline, we wrap the workflow —
from requirements extraction through retrieval — into a structured **DataOps DAG** using the
**Skrub** framework. As shown in the computation graph, the pipeline maps variables like
`dataset_id` and `subset` into functions such as `load_dataset`, runs `dataset_statistics` and
`image_metadata` as **independent parallel branches**, and finally converges at `merge_analysis`.
While this Skrub DAG is currently streamlined around core data operations, it is fully extensible
and will scale natively as we integrate our upcoming **Data Augmentation** module.
